\chapter{Evaluation}
This chapter evaluates the implementation presented in this thesis. It describes the experimental setup, reports the results obtained from analysing the selected subject systems, and discusses threats to the validity of the evaluation. In addition, the findings are reflected upon in a subsequent discussion, where the results are interpreted, limitations are considered, and the strengths and weaknesses of the proposed approach are examined.
\section{Setup}
% What repositories have you examined?
The evaluation is conducted on a set of real-world, configurable software systems that use preprocessor-based variability. These systems constitute the subject systems of the study. The selection of subject systems was performed incrementally. Initially, smaller repositories were chosen in order to validate the analysis workflow and to assess the behaviour of the implementation on a manageable commit history.
As an initial subject system, the repository \texttt{xterm} was selected. The analysed revision range of \texttt{xterm} contains 112 commits and exhibits non-trivial use of conditional compilation. This limited size made it possible to inspect the analysis results in detail and to validate the correctness of the implementation before scaling to larger systems. After successfully analysing \texttt{xterm}, additional subject systems with larger commit histories were considered. In particular, the repository \texttt{libssh} was analysed, which contains a substantially larger number of commits and more complex variability-related code. This allowed the evaluation to assess the behaviour of the implementation on a more realistic and demanding subject system.

Each subject system was cloned locally and analysed using \texttt{DiffDetective} in combination with the custom analysis component \texttt{MyAnalysis}. The repositories were provided to DiffDetective as input and processed commit by commit. For each pair of successive commits, DiffDetective computed a variability-aware diff, capturing changes to annotated source code together with their associated presence conditions. \texttt{MyAnalysis} was executed as part of this traversal and recorded commit-level metadata and classification results.

The evaluation focuses on measuring variability-related changes at commit level. For each analysed commit, the implementation records the assigned variability edit class, as provided by DiffDetective, and whether the commit message indicates a potential bug fix based on a keyword-based heuristic. These measurements support both quantitative analysis, such as counting the frequency of different edit classes, and qualitative analysis, such as inspecting individual commit messages and corresponding code changes.

To support higher-level interpretation of the results, a topic analysis was applied as a post-processing step. The topic analysis focuses specifically on commits classified as \emph{Refactoring} or \emph{Reconfiguration}, as these represent higher-level changes to variability. A lightweight keyword-frequency approach was used to analyse the commit messages associated with these commits, allowing recurring terms and themes to be identified without relying on complex semantic models.

\section{Results}
% What were the results
This section presents the results of applying the proposed analysis to the selected subject systems. 
The results are reported both quantitatively and qualitatively, focusing on 
(1) how often different types of variability-related changes occur, 
(2) how frequently commits are classified as bug-related, and 
(3) which recurring themes can be observed through topic analysis of commit messages. 
In addition, selected commits are inspected manually to validate and interpret the automated results 
and to illustrate how variability-related changes manifest in practice.

\subsection{Overall Results}

Overall, the analysis demonstrates that variability-aware commit classification can be combined with commit-message inspection to study how presence conditions evolve in practice. Using DiffDetective together with MyAnalysis, commits that modify presence conditions are identified and classified according to their variability semantics, enabling higher-level distinctions such as Refactoring and Reconfiguration. This provides a basis for relating structural variability edits to potential developer mistakes that are self-reported through commit messages.

A key observation is that the effectiveness of self-reported bug detection depends heavily on how developers write commit messages. In the analysed revision range of xterm, commit messages are typically short and identifier-like, as illustrated in Figure~\ref{fig:commit-message-styles}. The keyword-based heuristic therefore flags no commits as bug-related. In contrast, in the analysed subset of libssh, commit messages frequently contain explicit corrective terms (e.g., ``fix'', ``bug'', ``error''), also illustrated in Figure~\ref{fig:commit-message-styles}, and many commits are consequently classified as bug-indicating. This difference suggests that the absence of bug-indicating keywords does not imply the absence of variability-related issues but rather reflects how developers document changes.

In addition, the results indicate that higher-level variability edits often occur in a corrective maintenance context in actively maintained systems. Manual inspection of selected commits supports this interpretation: bug-indicating commits classified as Refactoring or Reconfiguration frequently involve adjusting \#if/\#ifdef logic, , fixing incorrect or misplaced guards, or correcting configuration-specific behaviour. At the same time, real-world preprocessor code is not always well-structured (e.g., unusual formatting or incomplete directive blocks). This causes parsing and annotation errors and means that some commits cannot be analysed reliably. The following subsections report the quantitative distributions and the qualitative observations in detail.

\subsection{Quantitative Results}
This section reports the quantitative results produced by running DiffDetective together with MyAnalysis on the selected subject systems. The measurements are derived from the per-commit output logs generated during the analysis (commit classifications and commit-messages). Quantitatively, the evaluation focuses on three aspects: (1) how frequently presence condition changes occur, (2) how these changes are distributed across DiffDetective’s variability edit classes (in particular Refactoring and Reconfiguration), and (3) how often commits with higher-level variability edits overlap with bug-indicating commit messages.
For xterm, the analysis recorded 111 commits in the produced logs. Across these commits, none were classified as bug-related based on the keyword heuristic applied to commit messages. This result is consistent with the commit-message style in xterm, where messages are typically short, identifier-like (Figure~\ref{fig:commit-message-styles}) , and contain little descriptive information. As a consequence, commit-message-based bug detection yields no matches even if variability-related changes are present, illustrating that quantitative results based on self-reported intent depend strongly on how developers document their changes.
For libssh, DiffDetective traversed the repository history (5,352 commits), but the quantitative dataset available for evaluation is limited to the subset of commits that remained in the final per-commit logs. Because the analysis was executed multiple times with multi-threading enabled and output files reset during initialisation, the resulting log files may include only part of the traversed history. Therefore, counts computed for libssh are interpreted as measurements on the recorded subset, rather than absolute frequencies over the full repository history. Despite this limitation, the available logs show a clear tendency towards corrective maintenance: many commit messages explicitly include bug-related terms such as fix, bug and error, and the heuristic correspondingly flags a portion of the recorded commits as bug-related.
To connect variability edits to developer-reported mistakes, the key quantitative outcome is the overlap between Refactoring/Reconfiguration classifications and bug-indicating commit messages. In the recorded subset of libssh commits, a noticeable share of commits classified as Refactoring or Reconfiguration is also flagged as bug-related by the commit-message heuristic. This provides evidence that higher-level variability edits changes that restructure or adjust existing presence conditions rather than merely adding or deleting lines, frequently occur in the context of corrective intent, supporting the thesis focus on variability-related mistakes observable at the level of variability semantics.
In addition to commit-level classifications, the analysis reports quantitative indicators of irregular variability usage through parsing and annotation errors encountered when extracting presence conditions from real-world codeFor libssh, the exported metadata reports line-continuation errors, unclosed annotations, and cases where extracted presence-condition formulas could not be parsed. These results quantify the extent to which preprocessor usage in practice can violate idealised assumptions of well-formed annotations, which impacts how reliably commits can be analysed and is therefore an important quantitative limitation of the evaluation.
Overall, the quantitative results show that (i) the distribution and interpretability of bug-related commits depends strongly on repository-specific commit-message practices (as seen in the contrast between xterm and libssh, Figure~\ref{fig:commit-message-styles}), (ii) in libssh, many recorded commits reflect corrective intent, and (iii) within the recorded data, higher-level variability edits (Refactoring/Reconfiguration) frequently coincide with bug-indicating commit messages, suggesting a recurring link between presence condition evolution and developer-reported mistakes.

\subsection{Qualitative Results}
This subsection complements the quantitative measurements with a qualitative inspection of selected commits. The goal is; firstly, to validate that the automated classifications produced by DiffDetective and MyAnalysis correspond to the actual changes in the code, and second, to illustrate how variability-related changes appear in practice when developers modify presence conditions. The qualitative analysis therefore focuses on commits classified as Refactoring or Reconfiguration, and in particular on those whose commit messages indicate bug-fixing intent (self-reported bugs).
Across the inspected commits, the commit messages generally provide useful context in repositories where developers describe changes in natural language, as observed for libssh(Figure~\ref{fig:commit-message-styles}). For example, messages such as ``Fix the ifdefs.'' align closely with the corresponding DiffDetective classification of Reconfiguration (Figure~\ref{fig:bug-related-reconfiguration}), which indicates that an existing presence condition was adjusted rather than new variability being introduced. Manual inspection of these commits confirms that the changes are typically small in terms of raw text, but semantically meaningful in terms of the configuration space affected by conditional compilation (cite Apel featured..). In practice, the inspected reconfiguration commits often modify the structure of \#if/\#ifdef logic, e.g., by correcting a condition, moving a guard or updating which configurations include a code block.
A recurring observation is that many bug-fixing commits correct misconfigurations rather than fixing a traditional bug in one specific configuration.
Several commits fix conditions that were either too broad (including code in configurations where it should be excluded) or too narrow (excluding code where it is required), and these mistakes are often discovered only after a build failure or a platform-specific problem that only appears under certain configurations. In such cases, the variability-aware diff helps make the semantic impact explicit: even though the textual edit may be limited to a single directive, it changes the presence condition under which code is compiled, which can affect multiple variants at once.
In addition to reconfigurations, commits classified as Refactoring often represent changes where developers restructure conditional compilation without intending to change functionality. The inspected refactoring commits frequently involve moving preprocessor directives, restructuring multi-level conditions or simplifying the layout of variability annotations. While these edits are typically motivated by maintainability, they can still introduce mistakes when guards are moved to an incorrect scope or when the refactoring unintentionally alters the logical condition. In the inspected sample, this is reflected by cases where a commit is classified as Refactoring, yet the commit message still signals corrective intent, suggesting that refactoring and bug fixing can be intertwined when variability logic is complex.
The qualitative inspection also highlights limitations of the commit-message heuristic. Some commits flagged as bug-related appear to address issues such as build warnings, code cleanup or portability adjustments rather than definitive runtime faults. However, these commits still represent corrective intent in the sense that developers explicitly describe them as fixing something and the associated presence-condition edits can be relevant for understanding how developers correct variability-related problems over time. For this reason, the qualitative analysis interprets ``bug-related'' as an indicator of developer-reported correction rather than a precise fault class.
Finally, inspecting commits that triggered parsing or annotation errors shows that real-world preprocessor code is not always written in a clean and consistent way. For example, unusual formatting, commented-out directives or long directives split across multiple lines can make it difficult to extract presence conditions reliably. As a result, some commits cannot be classified automatically, which highlights a practical limitation of variability-aware analysis on real repositories.
Overall, the qualitative results support the main quantitative findings: variability edits (Refactoring and Reconfiguration) occur in real systems and often overlap with developer-reported corrective intent. The manual inspection confirms that these commits frequently involve adjustments to preprocessor guards and presence-condition logic that have non-local semantic effects across configurations, illustrating how developer mistakes related to variability can arise and how they are subsequently corrected in practice.

\subsection{Topic Analysis of Commit Messages}
To support interpretation of the variability-related edits, a lightweight keyword-frequency analysis was applied to commit messages associated with commits classified as \emph{Refactoring} or \emph{Reconfiguration}. The purpose of this step is not to derive detailed semantic topics, but to provide a simple overview of which activities and contexts are mentioned most often in commit messages when developers modify presence conditions.
The resulting keyword frequencies (Figure~\ref{fig:keyword-frequency}) show that variability-related commits are often described using corrective and configuration-oriented terms. Frequent tokens such as \emph{fix} and references to conditional compilation (e.g., \emph{ifdef}) suggest that presence-condition changes commonly occur in the context of maintenance and adjustment of configuration-dependent behaviour. Overall, the topic analysis provides supporting evidence that refactoring and reconfiguration edits are frequently accompanied by commit messages that emphasise correction and configuration management rather than feature addition.

\subsection{Error-Related Changes}

In addition to classifying variability-related changes, the evaluation analyses how often commits are associated with errors and how such errors manifest in practice. Error-related commits are identified using a lightweight, keyword-based heuristic applied to commit messages, as described in Chapter~3. A commit is considered error-related if its message contains commonly used terms such as \emph{bug}, \emph{fix}, \emph{error}, \emph{issue}, or \emph{defect}. Following prior work on self-admitted issues, this heuristic is interpreted as an indicator of corrective intent rather than a definitive classification of functional defects.

In addition to error-related commits inferred from commit messages, the analysis also reports errors that occur during the parsing of annotated source code and presence conditions. These parsing errors indicate situations in which real-world preprocessor usage violates assumptions of a well-formed annotation structure. For \texttt{libssh}, the exported metadata reports 97 occurrences of line-continuation errors, 34 cases of unclosed annotations, 4 occurrences of \texttt{\#else} or \texttt{\#elif} without a matching \texttt{\#if}, and 6 cases where the extracted presence-condition formula could not be parsed. Similar parsing issues are observed in \texttt{xterm}, where a high number of unclosed annotations and line-continuation errors occur despite the repository’s relatively small size.

These errors indicate that conditional compilation in real-world systems is often irregular and shaped by build-system constraints rather than strictly adhering to idealised preprocessor usage. Qualitative inspection shows that such cases typically arise from complex commented-out directives, or unconventional formatting. As a result, not all commits can be analysed reliably, and parsing errors represent both a limitation of the analysis and empirical evidence of variability-related complexity.

\paragraph{Quantitative Results}
The analysed subject systems differ strongly with respect to error-related changes. In \texttt{libssh}, a substantial proportion of the analysed commits are classified as error-related based on commit-message keywords. This indicates that corrective maintenance is a frequent activity in the evolution of variability-related code. Many of these commits are associated with refactoring or reconfiguration edits, suggesting that higher-level changes to presence conditions are often motivated by the need to correct incorrect or incomplete configurations rather than by feature evolution alone.

In contrast, \texttt{xterm} exhibits no commits that are classified as error-related based on commit-message analysis, despite the presence of numerous parsing and annotation-related issues. This discrepancy highlights that the absence of explicit error-related keywords in commit messages does not imply the absence of underlying problems. Instead, it reflects differences in developer communication practices and conventions regarding how issues are documented.

\paragraph{Qualitative Inspection}
To complement the quantitative analysis, selected error-related commits were inspected manually. In most cases, the commit message accurately reflects the nature of the change, and the corresponding variability-aware diff confirms that the commit addresses an actual defect. Typical examples include correcting incorrect \texttt{\#ifdef} conditions, adding missing error checks under specific configurations, or fixing platform-dependent compilation paths.

A recurring pattern is that many error-related commits address misconfigurations introduced by earlier changes. Edits to conditional compilation directives may appear local but can affect a large portion of the configuration space. As a result, developers often discover unintended effects only after the fact, leading to subsequent corrective commits. This explains why similar corrective patterns recur over time.

At the same time, some commits flagged as error-related primarily address maintainability concerns, such as build warnings or code cleanup, rather than functional defects. These cases illustrate that commit-message-based classification tends to over-approximate error-related changes and should be interpreted as an indicator of corrective intent rather than a precise fault classification.

Overall, the results show that error correction is a central driver of variability evolution in actively maintained systems such as \texttt{libssh}. By combining variability-aware structural analysis with commit-message inspection, the evaluation captures both explicit and implicit error-related activities.


\subsection{Threats to Validity}
\cite{Wohlin2024_experimentationInSE}

The evaluation is subject to several threats to validity. External validity is limited by the selection of subject systems. Only two repositories were analysed, and both are implemented in C using preprocessor-based variability. While these systems are representative of a common class of configurable software, the results may not generalise to other languages or variability mechanisms.

Scalability also poses a limitation. For large repositories with extensive commit histories, repeated initialisation and output overwriting during multi-threaded execution reduced the number of commits available for quantitative analysis. As a result, some findings are interpreted qualitatively rather than as absolute frequencies. Analysing larger numbers of systems and improving result aggregation are considered out of scope for this thesis.

Internal validity is affected by the use of keyword-based heuristics for identifying error-related commits. While manual inspection suggests that many classifications are accurate, the heuristic may both miss error-related commits and incorrectly flag maintenance-oriented changes as bug-related. This threat is mitigated by combining commit-message analysis with variability-aware structural classification.

\subsection{Discussion}

The evaluation demonstrates that variability-related changes are closely tied to error correction and maintenance activities. In particular, the results show that developer mistakes in conditional compilation are not necessarily frequent in terms of explicit documentation, but they can be severe in impact, affecting large parts of the configuration space and requiring subsequent corrective commits.

The combination of variability-aware differencing and commit-message analysis provides a more transparent view of how presence conditions evolve over time. While commit messages alone cannot fully explain variability evolution, they provide valuable contextual information when interpreted alongside structural edit classifications.

Future work could extend this approach to additional subject systems, improve scalability of result collection, or incorporate more advanced NLP techniques. Moreover, integrating variability-aware analysis directly into development tools could help developers better anticipate the semantic impact of changes to conditional compilation and reduce the likelihood of configuration-dependent errors.
